# Design Decisions

## Storage: localStorage with swap-ready interface

`taskStorage.ts` exports a `TaskStorageService` interface and a `localStorageService` implementation. The active service is assigned to `taskStorage` as a single export.

**To migrate to an API:** create `apiStorageService: TaskStorageService` and change the last line of `taskStorage.ts`. No other file needs to change.

**Current behaviour:** every React state update triggers a `useEffect` that calls `taskStorage.saveTasks(tasks)`, serialising the full task array to JSON.

---

## State: React Context, not a global store library

Task state lives in `TaskProvider` (React Context + `useState`). There is no Redux, Zustand, or Jotai. All mutations go through `useCallback`-wrapped handlers exposed via context.

`@tanstack/react-query` is installed (scaffolding default) but is not used for task data.

---

## Order field strategy

- **New task:** `order = Date.now()` — guarantees new tasks sort to the bottom of a column.
- **After reorder / cross-column move:** `order` is reassigned as the array index (0, 1, 2 …).
- Implication: after any drag, orders become small integers. New tasks created after a drag will have large `Date.now()` values, placing them at the bottom regardless of column size.

---

## `updateTask` type inconsistency

The `TaskContextValue` interface declares:

```typescript
updateTask: (id: string, updates: Partial<Pick<Task, "title" | "description" | "priority">>) => void;
```

The implementation narrows to `title | description` only:

```typescript
const updateTask = useCallback((id: string, updates: Partial<Pick<Task, "title" | "description">>) => { … })
```

`KanbanBoard` passes `priority` in the update object. This works at runtime (spread includes any key), but TypeScript does not enforce priority updates through the implementation signature. Strict mode is off, so no error is raised.

---

## TaskDialog: plain useState, not React Hook Form

`react-hook-form` is listed as a dependency but `TaskDialog.tsx` manages form state with four `useState` calls. This is intentional simplicity for the current form complexity (2 inputs + 2 selects).

---

## Animation approach: per-card Canvas elements

Rather than CSS-only animations, visual effects (black hole, portal, big bang, drag particles) are rendered on `<canvas>` elements mounted inside each `TaskCard`. Each canvas:
- Mounts when its trigger condition becomes true
- Runs its own `requestAnimationFrame` loop
- Calls an `onComplete` callback (for delete/portal) to fire the actual state mutation after the animation finishes
- Unmounts automatically via React state

This defers the destructive state changes (delete, move) until the animation completes, keeping the card visible during the effect.

---

## UI language: Portuguese (pt-BR)

All user-facing text (button labels, dialog titles, confirmation messages) is written in Brazilian Portuguese. Priority labels: Baixa / Média / Alta / Crítica.

---

## TypeScript: strict mode off

`tsconfig.app.json` has `"strict": false` and `"noImplicitAny": false`. Type safety is present but not enforced comprehensively.

---

## shadcn/ui: do not hand-edit

`src/components/ui/` is generated by `bunx shadcn@latest add <component>`. Edit via regeneration, not directly.

---

## `userId` field: placeholder for future multi-user

`Task.userId?: string` is declared in the type and persisted in localStorage, but is never populated by any code path. Reserved for a future authenticated multi-user implementation.
